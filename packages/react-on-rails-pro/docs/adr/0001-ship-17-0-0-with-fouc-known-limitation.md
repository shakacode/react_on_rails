# Ship 17.0.0 final with late-boundary FOUC as a documented known limitation

Verification against real builds (July 2026) showed that stock Shakapacker production builds are the primary silent-no-op case for the stream-transform CSS gating paths — Preload Promotion and Flight-Chunk Inference in `src/injectRSCPayload.ts`. Production builds emit numeric chunk ids and `css/[id]-<hash>.css` filenames, disabling Flight-Chunk Inference and Path A's old `css/clientN-*.css` filename fallback. Compatible or custom asset naming can still activate those paths, and dev/test named-chunk builds are not the same failure mode. In the verified production dummy captures, the only active protection for late boundaries was the Flight `preinit` hint path, which delivers CSS render-blocking-equivalently for boundaries resolved before shell flush but leaves a FOUC window for Suspense boundaries streamed after the shell (Fizz emits only a `preload`; the real stylesheet arrives at hydration).

We decided to ship 17.0.0 final anyway, documenting late-boundary FOUC as a known limitation, rather than blocking the release on a fix. Rationale: this is not a regression — every 17.0.0 RC user has been running hints-only protection, the RCs soaked without FOUC reports, and the fix requires stream-transform surgery (client-chunk CSS cannot be recognized from id-based filenames without manifest knowledge), not a config tweak. The fix — either manifest-driven matching or replacing the scraping design with React-native `$RR` gating — is the first priority for 17.0.x/17.1, with severity data gathered separately (throttled-browser repro on a delayed-boundary page).

Consequences: release notes and docs must disclose the limitation; the inert code paths stay in place for 17.0.0 (removing them is part of the fix work, not the release); unit-test fixtures must be reshaped to real-build asset names (prod numeric ids, `css/[id]-<hash>.css`) so an inert-matcher regression can never pass CI again.
