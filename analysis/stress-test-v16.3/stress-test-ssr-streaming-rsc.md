# Stress Test Report: SSR, Streaming, and RSC
**Date:** 2026-04-22
**Branch:** upcoming-v16.3.0
**App:** react_on_rails_pro/spec/dummy (Rails at localhost:3199, Node Renderer at localhost:3800)

---

## Executive Summary

All nine tested endpoints return HTTP 200, but **seven of nine pages silently fall back to client-side rendering** due to missing RSC manifest files. The server sends HTML that _looks_ correct (HTTP 200, full document) but embeds error templates that tell the browser to re-render on the client. Under 50-concurrent-request load, all requests succeed without corruption or timeouts.

---

## BUG 1 (Critical): RSC Manifest Files Missing -- All Streaming/RSC SSR Silently Falls Back to Client Rendering

**Severity:** Critical
**Affected pages:** 7 of 9 tested pages

### Reproduction

```bash
curl -s http://localhost:3199/stream_async_components | grep -o 'data-msg="[^"]*"'
```

### Response (in every streaming/RSC page)

```
data-msg="Switched to client rendering because the server rendering errored:

Error"
```

### Root Cause

The `react-client-manifest.json` and `react-server-client-manifest.json` files are not generated by the current webpack development build. The `RSCWebpackPlugin` from `react-on-rails-rsc` is configured in both `clientWebpackConfig.js` and `serverWebpackConfig.js`, but the manifest files are absent from all output directories:

```bash
# No manifests in ANY output directory:
find public/webpack/development -name 'react-*-manifest*'  # (empty)
find ssr-generated -name 'react-*-manifest*'                # (empty)
```

The node renderer logs the error on every render request:

```
[SERVER] Failed to load JSON file: .node-renderer-bundles/<hash>/react-client-manifest.json
{"errno":-2,"code":"ENOENT","syscall":"open","path":"...react-client-manifest.json"}
```

Old bundle hash directories DO contain these manifests (from a previous build), but current bundle hash directories do not.

### Impact

Every component that goes through the streaming/RSC pipeline (stream_react_component, RSC components, incremental rendering) renders an empty shell on the server and defers ALL rendering to the client. This defeats the purpose of SSR.

### Affected Pages

| Page | Status | Error |
|------|--------|-------|
| `/server_side_hello_world` | SSR works | Uses sync rendering, not RSC pipeline |
| `/native_metadata` | SSR works | Uses sync rendering, not RSC pipeline |
| `/stream_async_components` | HTTP 200, SSR FAILS | "Switched to client rendering...Error" |
| `/stream_async_components_for_testing` | HTTP 200, SSR FAILS | "Switched to client rendering...Error" |
| `/rsc_posts_page_over_http` | HTTP 200, SSR FAILS | "Switched to client rendering...Error" |
| `/rsc_echo_props` | HTTP 200, SSR FAILS | 6 components, all error |
| `/test_incremental_rendering` | HTTP 200, SSR FAILS | "Switched to client rendering...Error" |
| `/stream_native_metadata` | HTTP 200, SSR FAILS | "performance is not defined" (separate bug) |
| `/cached_stream_async_components_for_testing` | HTTP 200, SSR FAILS | "Switched to client rendering...Error" |

## BUG 2 (Medium): `performance` API Not Available in Node Renderer VM Context

**Severity:** Medium
**Affected page:** `/stream_native_metadata`

### Reproduction

```bash
curl -s http://localhost:3199/stream_native_metadata | grep -o 'data-msg="[^"]*"'
```

### Response

```
data-msg="Switched to client rendering because the server rendering errored:

performance is not defined"
```

### Root Cause

The `NativeMetadataApp` component uses `React.lazy` with a `setTimeout` wrapper. During SSR in the node renderer's VM sandbox, React's Suspense scheduler calls `performance.now()`, but the VM context created by `packages/react-on-rails-pro-node-renderer/src/worker/vm.ts` does not define the `performance` global. The VM provides polyfills for `setTimeout`, `setInterval`, etc. (lines 213-221) but omits `performance`.

### Stack trace (from SSR output)

```
ReferenceError: performance is not defined
    at lazyInitializer (webpack://react_on_rails_pro_dummy/...)
    at Suspense (<anonymous>)
    at NativeMetadataApp (webpack://react_on_rails_pro_dummy/./client/app/ror-auto-load-components/NativeMetadataApp.jsx?:91:3)
```

### Suggested fix

Add `performance` to the VM context in `vm.ts`:

```typescript
vm.runInContext(`var performance = { now: function() { return Date.now(); } }`, context);
```

## BUG 3 (Low): React Router Returns HTTP 200 for 404 Pages

**Severity:** Low

### Reproduction

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:3199/react_router/nonexistent_page_that_doesnt_exist
```

### Response

```
200
```

The response body contains "404 Not Found" and React Router's default error boundary message, but the HTTP status code is 200. This is a known React Router SSR limitation -- the router runs on the client and can't set the HTTP status code during SSR.

## Node Renderer Direct Testing

The node renderer at port 3800 uses a custom length-prefixed binary protocol, NOT HTTP REST. Direct curl testing is not possible. Authentication, protocol version checking, and bundle management all happen through this binary protocol. The Rails app communicates with it via the HTTPX HTTP/2 client library.

Notable node renderer log entries:
- `"No targetBundles provided. As of protocol version 2.0.0, targetBundles is required."` -- appears during initial bundle setup
- `"Worker 6 died UNEXPECTEDLY :("` -- a worker died and was restarted (occurred once)
- `"ERROR: filename param not provided to GET /asset-exists"` -- malformed request during startup
